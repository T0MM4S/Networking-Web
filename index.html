🌐 Leksion 1: Hyrje në Rrjeta Komunikuese
Definicione Kyçe

Çfarë është një Rrjet Komunikues?

Rrjet Komunikues: Pajisjet (hardware & software) dhe infrastruktura që ofrojnë shërbim bazë komunikimi. Zakonisht paraqitet si "cloud" për shkak se është transparent ndaj përdoruesit.
Pajisjet kryesore: Routers, servers, switches, multiplexers, hubs, modems
Fasilitetet: Tela bakri, kabllo koaksiale, fibra optike, ndenjëse, shtylla telefonie
Historia e Rrjetave

Evoluimi i Sistemit Telefonik

  • 1875 — Alexander Graham Bell punon mbi telegrafi harmonik, zbulon transmetimin e zërit
  • 1876 — Patenta e telefonit
  • 1877 — Themelimi i Bell Telephone Company
  • 1877 — Shpikja e switch-it me patchcord (Circuit Switching)
  • 1888 — Strowger shpik switch-in automatik
  • 1961 — T-1 multiplexing: 24 zëra = 1.544 Mbps
  • 1980s — Digital Switching
  • 1990s — Digital Cellular, Fiber Optike (OC-192 = 10 Gbps)
Problemi N²: Për N përdorues → nevojiten N(N-1)/2 lidhje direkte
Zgjidhja: Circuit Switching → vetëm N lidhje me qendrën
Historia e Internetit

Evoluimi i Rrjetave Kompjuterike

  • 1950s — Teknologjia telegrafe adaptuar për kompjuterë
  • 1960s — Terminale "bosh" aksesin server-in qendror (SABRE)
  • 1970s — ARPANET (packet switching), TCP/IP, Ethernet LAN
  • 1973 — Martin Cooper (Motorola) shpik telefonin mobil
  • 1974 — Protokolli TCP (Cerf & Kahn)
  • 1985-1995 — NSFNET si backbone i Internetit
  • 1979-1983 — 1G: Rrjeta analoge mobile (Japoni, USA)
  • 2G — Dixhital: GSM (Europë), CDMA (USA), SMS
  • 3G — Circuit + Packet switching
  • LTE/4G — Vetëm IP packet switching, deri 1 Gbps
  • 5G — 10x bitrate, 10x latency↓, 100x kapacitet ndaj 4G

mMTC (Massive Machine Type Comm.) — Sensing, metering, monitoring

Brezat: FR1 (450 MHz–6 GHz) dhe FR2 (24 GHz–52 GHz)

📦 Leksion 2: Application Layer
Arkitektura

Client-Server

  • Server: gjithmonë aktiv, IP e përhershme
  • Klientët NUK komunikojnë direkt me njëri-tjetrin
  • Shembuj: HTTP, IMAP, FTP
  • Zakonisht në data centers
Arkitektura

Peer-to-Peer (P2P)

  • Asnjë server gjithmonë aktiv
  • Peer-ët kërkojnë dhe ofrojnë shërbime njëri-tjetrit
  • Self-scalable — peer-ët e rinj sjellin kapacitet të ri
  • IP dinamike, menaxhim kompleks
  • Shembull: BitTorrent

Sockets dhe Adresimi

Socket — Ndërfaqja midis shtresës aplikacioni dhe shtresës transport brenda një hosti. Quhet edhe API (Application Programming Interface). Si "dera" — mesazhi del nëpërmjet tij.
Identifikuesi i procesit: Adresa IP (32-bit) + Port Number. Shembull: HTTP port 80, Mail port 25
Identifikuesi: IP_Address : Port_Number
Shembull: 128.119.245.12:80

Protokolli HTTP

HyperText Transfer Protocol

HTTP — Protokolli i shtresës aplikacioni për Web. Përdor TCP (port 80). HTTP është stateless — serveri nuk mban gjendjen e klientit.

Non-Persistent HTTP

  • Çdo objekt kërkon lidhje TCP të re
  • Koha = 2RTT + koha e transmetimit
  • Overhead i OS për çdo lidhje
Response time = 2RTT + file_transmission_time

Persistent HTTP (HTTP 1.1)

  • Lidhja TCP mbetet hapur pas përgjigjes
  • Shumë objekte mbi 1 lidhje TCP
  • Minimum 1 RTT për të gjithë objektet
  • Gjysma e kohës krahasuar me non-persistent

HTTP Request Methods

GET — Kërkon objekt nga serveri
POST — Dërgon të dhëna nga forma në body
HEAD — Kërkon vetëm headers (pa body)
PUT — Ngarkon file/objekt të ri në server

HTTP Response Status Codes

200 OK — Sukses, objekti vijon
301 Moved Permanently — Objekti lëviz, lokacioni i ri jepet
304 Not Modified — Conditional GET, cache valid
400 Bad Request — Mesazh i pakuptueshëm
404 Not Found — Dokumenti nuk u gjet
505 HTTP Version Not Supported

Cookies — Ruajtja e Gjendjes

HTTP është stateless, por Cookies lejojnë ruajtjen e gjendjes. Katër komponentët:
  • Cookie header në HTTP response message (serveri → klient)
  • Cookie header në HTTP request message (klient → server)
  • File cookie i ruajtur tek hosti i klientit (menaxhohet nga browser)
  • Database backend tek website-i
Third-party cookies (tracking) — Lejojnë gjurmimin e sjelljes mbi shumë faqe. Disabled by default në Firefox, Safari. GDPR (2018) rregullon këtë.

Web Caches (Proxy Servers)

Web Cache — Satisfaton kërkesat e klientëve pa e përfshirë serverin origjinal. Vepron si si klient dhe si server njëkohësisht.
  • Redukton kohën e përgjigjes (cache është afër klientit)
  • Redukton trafikun në access link të institucionit
Shembull: Hit rate 0.4 → access link utilization ↓ nga 0.97 në 0.58
Average delay ↓ nga "minutes" në ~1.2 sec
Conditional GET — If-modified-since header. Nëse objekti nuk ka ndryshuar → 304 Not Modified (asnjë transmetim objekti!)

HTTP/2 dhe HTTP/3

HTTP/1.1 — HOL (Head-of-Line) blocking: objektet e vogla presin prapa objekteve të mëdha.
HTTP/2 (RFC 7540, 2015) — Objektet ndahen në frames, frames transmetohen të ndërmishëm. Prioritet i bazuar në preferencat e klientit. Push objekte pa kërkesë.
HTTP/3 — Shton siguri dhe kontroll errori/congestion per-objekt mbi UDP. Zgjidh problemin e bllokimit nga humbja e paketave.

Protokolli DNS

Domain Name System

DNS — Bazë të dhënash e shpërndarë implementuar si hierarki e serverëve të emrave. Protokoll i shtresës aplikacioni. Port 53 (UDP dhe TCP).

Shërbimet DNS

  • Hostname-to-IP-address translation
  • Host aliasing (emra alternativë)
  • Mail server aliasing
  • Load distribution (shumë IP → 1 emër)

Hierarkia e Serverëve DNS

Root Name Servers — 13 serverë logjikë botërorë (replicated ~200 në USA). Menaxhohen nga ICANN. DNSSEC siguron autentifikimin.
TLD Servers (Top-Level Domain) — Responsible për .com, .org, .edu, .net, etj. dhe domaine kombëtare (.al, .uk, .de)
Authoritative DNS — Serverët DNS të organizatës, ofrojnë hostname → IP për hostet e organizatës
Local DNS — Çdo ISP ka nga 1. Hosti dërgon query-n te Local DNS i tij. Port 53 UDP.

Iterated Query DNS

  • Local DNS → Root → (emrin e TLD)
  • Local DNS → TLD → (emrin e Authoritative)
  • Local DNS → Authoritative → IP
  • Local DNS → klienti

✓ Best practice — ulet ngarkesa tek Root/TLD

Recursive Query DNS

  • Hosti → Local DNS
  • Local DNS → Root
  • Root → TLD (recursive)
  • TLD → Authoritative (recursive)

🔗 Leksion 3: Transport Layer
Transport Layer — Ofron komunikim logjik midis proceseve aplikacioni që punojnë në hose të ndryshëm. Dërgues: ndan mesazhet në segmente → shtresës rrjetit. Marrësi: rimonon segmentet → shtresës aplikacioni.
Dallimi Transport vs Network Layer: Network layer — komunikim host-me-host. Transport layer — komunikim proces-me-proces (bazuar në shërbimin e shtresës rrjetit).

TCP — Transmission Control Protocol

Reliable, in-order delivery

Flow control

Congestion control

Connection setup

Full duplex

Cumulative ACKs

Nuk garanton: timing, minimum throughput, security

UDP — User Datagram Protocol

Unreliable, unordered

Pa flow control

Pa congestion control

Pa connection setup

Shpejtësi

Checksum (detektim gabimesh)

Përdoret: DNS, SNMP, HTTP/3, streaming, gaming

Multiplexing / Demultiplexing

Multiplexing (dërgues) — Mbledh të dhënat nga sockets të shumta, mbështjell me header → segmente.
Demultiplexing (marrës) — Dërgon segmentin e marrë tek socketi i duhur duke përdorur IP/port.

UDP Demultiplexing

Bazuar vetëm në destination port number. Datagrams me dest port të njëjtë por source të ndryshme → i njëjti socket.

TCP Demultiplexing

Bazuar në 4-tuple: Source IP + Source Port + Dest IP + Dest Port. Secili socket ka 4-tuple unike.

UDP Segment Structure

| source port # | dest port # | (32 bits) | length | checksum | | application data |
Length — Gjatësia totale e segmentit UDP (header + data) në bytes.
Checksum — Detekton gabime (bit të flipuar). One's complement sum i 16-bit integers. Nëse jo i barabartë → gabim i zbuluar.
Reliable Data Transfer

Principet e RDT

rdt1.0 — Kanal i besueshëm+
Kanali nuk ka gabime, nuk humbet pako. Dërgusi dërgon, marrësi pranon. Asnjë mekanizëm i posaçëm i nevojshëm.
rdt2.0 — Kanal me bit errors+
Checksum detekton gabime. Mekanizmat: ACK (OK) dhe NAK (gabim → retransmit). Defekti fatal: Nëse ACK/NAK dëmtohet, dërgusi nuk di çfarë ndodhi!
rdt2.1 — Sequence Numbers+
Shton sequence numbers për të trajtuar ACK/NAK të dëmtuar. 2 seq # (0,1) mjaftojnë. Marrësi hedh poshtë kopjet e dyfishta.
rdt2.2 — NAK-free+
Vetëm ACK, pa NAK. Marrësi ACK-on pakon e fundit të marrë OK (specifikon seq #). ACK i dyfishtë = NAK. TCP përdor këtë qasje.
rdt3.0 — Kanal me gabime dhe humbje+
Kanalë i humb pako (data dhe ACK). Qasja: dërgusi pret "kohë të arsyeshme" për ACK → timer countdown. Nëse skadoi → retransmit. Seq # trajton duplikatet.

Stop-and-Wait vs Pipelining

Stop-and-Wait — Dërgon 1 paketë, pret ACK. Performancë shumë e dobët!
Shembull: 1 Gbps link, 15ms propagation, 8000-bit packet
Utilization = (L/R) / (RTT + L/R) ≈ 0.00027 = 0.027%
Pipelining — Dërgues lejon shumë pako "in-flight" pa ACK. Rrit utilizimin drastikisht. Kërkon: seq numbers të zgjeruara, buffering.

Go-Back-N (GBN)

  • Window deri N pako pa ACK
  • Cumulative ACK(n) — ACK të gjitha deri n
  • Timeout(n) → retransmit n DHE të gjitha pas tij
  • Marrësi hedh poshtë out-of-order pako
  • Timer vetëm për pakon më të vjetër

Selective Repeat (SR)

  • Marrësi ACK-on çdo paketë individualisht
  • Buffer-on pako jashtë rendit
  • Dërgusi ka timer për çdo paketë unACKed
  • Timeout → retransmit VETËM atë paketë
  • Window size ≤ seq_space / 2 (për të shmangur konfuzionin!)
TCP në Detaje

TCP — Struktura dhe Funksionimi

MSS (Maximum Segment Size) — Madhësia maksimale e segmentit.
Sequence Number — Byte-stream numri i bajtit të parë të segmentit.
ACK Number — Seq # i bajtit tjetër të pritur nga ana tjetër. Cumulative ACK.
TCP Segment Header (fushat kryesore): - Source/Dest Port: 16 bit secili - Sequence Number: 32 bit - ACK Number: 32 bit - Header Length: 4 bit - ACK bit, RST, SYN, FIN bits - Receive Window (rwnd): 16 bit ← flow control - Checksum: 16 bit

TCP Round Trip Time (RTT) dhe Timeout

EstimatedRTT = (1-α)·EstimatedRTT + α·SampleRTT [α = 0.125] DevRTT = (1-β)·DevRTT + β·|SampleRTT - EstimatedRTT| [β = 0.25] TimeoutInterval = EstimatedRTT + 4·DevRTT
EWMA (Exponential Weighted Moving Average) — Influenca e mostrave të vjetra zvogëlohet eksponencialisht. Smooth-on variacionet.

TCP Flow Control

Flow Control — Pengon dërguesin të "overflowojë" buffer-in e marrësit. Marrësi reklamojë rwnd (receive window / free buffer space) në header TCP.
Dërgusi limits: unACKed data ≤ rwnd RcvBuffer default: 4096 bytes (OS auto-adjust)

TCP 3-Way Handshake

Para shkëmbimit të të dhënave, dërgues/marrës "handshake-ojnë" për të rënë dakord mbi parametrat (ISN — Initial Sequence Number).
Hapi 1: Client → Server: SYN (SYN bit=1, seq=x) Hapi 2: Server → Client: SYNACK (SYN=1, ACK=1, seq=y, ack=x+1) Hapi 3: Client → Server: ACK (ACK=1, seq=x+1, ack=y+1)
Mbyllja e TCP: FIN bit=1. Secili anë mbyll vete. FIN → ACK. Mund të kombinohet FIN+ACK.
Pse jo 2-way handshake? Variablat e vonesa, retransmissions, reordering e bën të pasigurt.

TCP Fast Retransmit

Nëse dërgusi merr 3 ACK të dyfishta (triple duplicate ACK) → retransmit menjëherë para skadimit të timeoutit. Humbja e paketës detektohet shpejt.
🗺️ Leksion 4: Network Layer
Network Layer — Transmeton segmentin nga hosti dërgues tek hosti marrës. Dërgusi encapsulates segmentet në datagrams → link layer. Protokollet e shtresës rrjetit gjenden në çdo pajisje Internet: hosts, routers.
Forwarding — Lëviz paketat nga porta hyrëse e router-it tek porta dalje e duhur. (Trajtim i "një kryqëzimi")
Routing — Përcakton rrugën e paketave nga burimi tek destinacioni. (Planifikimi i udhëtimit)
IP Addressing

IP Address: Hyrje

IP Address — Identifikues 32-bit i lidhur me çdo ndërfaqe host/router. Shprehet në notacion dotted-decimal (p.sh. 223.1.1.1).
Ndërfaqe (Interface) — Lidhja midis host/router dhe link fizik. Router-ët kanë zakonisht shumë ndërfaqe.

Subnets dhe CIDR

Subnet — Ndërfaqe të pajisjes që mund të arrijnë njëra-tjetrën fizikisht pa kaluar nëpër router ndërmjetës. Pajisjet në të njëjtin subnet kanë high-order bits të njëjtë.
CIDR (Classless InterDomain Routing, "cider") — Pjesa e subnet mund të jetë me gjatësi arbitrare. Format: a.b.c.d/x ku x = # bits të subnet.
Shembull: 200.23.16.0/20 Subnet mask /24 = 255.255.255.0 (24 bits subnet, 8 bits host) 2 adresa rezervohen: all-zeros (subnet) dhe all-ones (broadcast)

DHCP — Dynamic Host Configuration Protocol

DHCP — Hosti merr IP dinamikisht nga serveri kur "bashkohet" rrjetit. "Plug-and-play". Lejon ripërdorimin e adresave.
4 Fazat DHCP: 1. DHCP Discover (broadcast nga klienti — opsional) 2. DHCP Offer (serveri ofron IP — opsional) 3. DHCP Request (klienti kërkon IP) 4. DHCP ACK (serveri konfirmon IP)
DHCP gjithashtu kthen: adresën e first-hop router, emrin dhe IP e DNS server, network mask.
DHCP mesazhi encapsuluar në UDP, encapsuluar në IP, encapsuluar në Ethernet, broadcast (FFFFFFFFFFFF).

NAT — Network Address Translation

NAT — Të gjitha pajisjet e rrjetit lokal ndajnë 1 adresë IPv4 publike ndaj botës së jashtme. Private IP space: 10/8, 172.16/12, 192.168/16.
  • 1 IP adresë publike nga ISP për të gjitha pajisjet
  • Mund të ndryshosh adresat lokale pa notifikuar botën
  • Siguri: pajisjet lokale nuk janë drejtpërsëdrejti të adresueshme
Dalje: (source IP, port) → (NAT IP, new port) Hyrje: (NAT IP, new port) → (source IP, port) [nga NAT table]
Kontroversë NAT: Routers "duhet" të procesojnë vetëm deri shtresën 3. Shkel argumentin end-to-end. Por: NAT është kudo (shtëpi, celulare, 4G/5G).

IPv6

Motivimi fillestar: 32-bit IPv4 address space do të shterrohej. IPv6: 128-bit space.
IPv6 Datagram Format (40-byte header fikse): - Version (4 bit) - Traffic class, Flow label - Payload length - Next header - Hop limit - Source/Destination address (128 bit secili) Mungon (ndaj IPv4): Checksum, Fragmentation, Options
Tunneling — IPv6 datagrams si payload brenda IPv4 datagrams kur kalon nëpër routers IPv4 ("paketë brenda paketës"). Adoptim: ~40% klientë Google (2023).

Generalized Forwarding & OpenFlow

"Match + Action" — Abstraktim: match bits në paketën hyrëse, ndërmerr veprim. Generalized: shumë fusha header mund të përcaktojnë veprimin.
Flow Table — Çdo router përmban. Entries: match pattern, priority, action (drop/forward/modify), counters.

Middleboxes

Middlebox (RFC 3234) — "Çdo kuti ndërmjetëse që kryen funksione jashtë funksioneve normale të IP router-it në rrugën e të dhënave midis hostit burimor dhe destinacionit."
  • NAT — shtëpi, celulare, institucionale
  • Firewall, IDS — korporata, ISP
  • Load balancers — korporata, datacenter
  • Caches — ISP, mobile, CDN
Trendi: nga hardware i mbyllur → "whitebox" me open API → SDN + NFV (Network Functions Virtualization)

Principet Arkitekturore të Internetit

"Qëllimi është lidhshmëria, mjet është IP protokolli, dhe inteligjenca është end-to-end e jo e fshehur brenda rrjetit." (RFC 1958)
  • Lidhshmëri e thjeshtë
  • IP si "ngushtësia e orës" (hourglass) — shtresa e vetme e pandryshuar

End-to-End Argument — Funksione të caktuara (transmetim i besueshëm, congestion) mund të implementohen brenda rrjetit OSE tek skajet. Diskutim i vazhdueshëm!

Checksum Demo

16-bit One's Complement (TCP/UDP Header)

Qëllimi: të shohësh qartë si llogaritet checksum me mbledhje 16-bit, wrap-around carry dhe komplement final.

Si ndodh i gjithë procesi?

  1. Paketa ndahet në fjalë 16-bit (header + data, me pad nëse duhet).
  2. Të gjitha fjalët mblidhen me one's complement addition.
  3. Nëse del carry nga biti i 17-të, carry kthehet mbrapsht te 16 bitët.
  4. Në fund bëhet invertimi bit-për-bit dhe ai rezultat vendoset si checksum.
  5. Marrësi e bën të njëjtën llogaritje; rezultat i saktë jep all-ones (validim).
